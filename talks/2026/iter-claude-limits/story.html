You're Not Hitting Claude's Limits. You're Just Using It Wrong.

A guide to workspace anatomy, context windows, and the token habits that make or break your AI workflow, explained for humans with things to do.

Soumendra Kumar Sahoo · Tips & Tricks for Claude · 3 parts · ~15 min read

Here is a scenario that happens to almost everyone. You open Claude, fire off a clever prompt, get a brilliant answer, and then, three days and forty messages later, Claude starts forgetting things, slowing down, or refusing to cooperate entirely. You blame the model. You blame Anthropic. You post a frustrated tweet.

But here's what nobody told you: Claude isn't broken. You just haven't learned how it thinks. Once you understand its anatomy, how it stores context, what "tokens" actually mean, and where its configuration lives, everything changes. This is that guide.

"Claude doesn't count messages. It counts tokens. And that one fact changes everything about how you should work with it."

Soumendra Kumar Sahoo

We'll cover three things. First, the vocabulary, the building blocks of the Claude ecosystem that most people never bother to learn. Second, the .claude/ folder, the hidden control centre that lets you configure Claude like a professional. Third, the token game, ten habits that will stop you from hitting usage limits and make your interactions dramatically more effective.

Part 0: Introductory Concepts

The Seven Words You Need to Know

Most people use Claude the same way they use a search engine: type something in, read something back, repeat. That works, until it doesn't. To really leverage Claude, you need to understand seven distinct concepts that form its ecosystem. Each one has a specific job, and confusing them is the single biggest source of frustration for new users.

Think of it like this. A kitchen has knives, chopping boards, pots, ovens, recipe books, and a head chef. They're all different things. Using a knife when you need a pot isn't a failure of the kitchen, it's a failure of understanding the tools.

1. Prompts, the conversation you're having right now

A prompt is the most basic unit of interaction. It's the message you type. It's ephemeral, it only exists for the current conversation. When the chat ends, the prompt is gone.

The key insight: If you find yourself typing the same thing at the start of every conversation, "act as a senior developer", "always use TypeScript", "keep responses concise", that's not a prompt problem. That's a sign you need one of the tools below.

2. Projects, your long-term workspace

Projects are self-contained workspaces with their own chat histories and a 200,000-token knowledge base. Anything you upload into a Project, documents, code, PDFs, becomes available across every conversation inside it. Claude caches this content, so it doesn't re-read it from scratch each time.

Use Projects when you're doing sustained, multi-session work on a specific topic. The signal that you should stop using a Project and use a Skill instead? When you find yourself copying the same instructions into multiple Projects.

3. Rules, targeted instructions for specific files

Rules solve a very specific problem: what do you do when you have instructions that only apply to some files? You don't want your API design conventions cluttering up the instructions for your frontend components.

Rules are path-scoped configuration files. Claude only loads them when you're editing files that match the defined pattern. Here's what one looks like:

---
paths:
  - "src/api/**/*.ts"
  - "src/handlers/**/*.ts"
---
# API Design Rules
- All handlers return { data, error } shape
- Never expose internal error details to clients

Claude reads this rule only when you're working inside those paths. Everywhere else, it's invisible. This is how you keep your main configuration file clean.

4. Skills, Claude's memory for how to do things

This is the concept most people miss, and it's the most powerful. Skills are packaged procedures, instructions, scripts, and supporting resources, that Claude discovers and loads automatically when a task matches.

The magic is in how they load. Claude reads the metadata first (about 100 tokens). If the task matches, it loads the full instructions (under 5,000 tokens). Only if it needs to does it load the bundled files. This is called progressive disclosure, you're not burning tokens on things Claude doesn't need.

When to create a Skill: The signal is simple. If you're typing the same prompt across multiple conversations, "here's how to review a PR", "here's our brand voice", "here's how to manipulate this Excel format", turn it into a Skill. Once. Use it forever.

5. Subagents, delegation with guardrails

Subagents are specialised AI assistants with their own context windows, custom system prompts, and specific tool permissions. They handle discrete tasks independently and return results to the main agent.

Think of them as trusted interns with a defined job description. You might have a "Code Reviewer" subagent restricted to read-only tools and the cheaper Haiku model, fast, focused, and incapable of accidentally deleting your database. Claude can delegate to subagents automatically based on their descriptions, or you can invoke them explicitly.

6. MCP, the universal power socket

MCP stands for Model Context Protocol. Think of it as the USB-C of AI integrations. It's an open standard that lets Claude connect to external systems: Google Drive, Slack, GitHub, databases, internal tools.

The distinction that matters: use MCP to give Claude access to tools and data. Use Skills to give Claude knowledge of how to use those tools or follow your procedures. They're complementary, not interchangeable.

Part 1: The .claude/ Folder

The Control Centre Nobody Shows You

There's a hidden folder in your project that most Claude users have never opened. It's called .claude/, and it's where the real configuration lives. Understanding it is the difference between using Claude like a tourist and using it like a local.

It comes in two flavours. The one inside your project (your-repo/.claude/) gets committed to Git, it's shared team configuration, project-specific rules, agents, and Skills. The one in your home directory (~/.claude/) stays on your local machine, personal preferences, global memory, cross-project Skills.

your-project/
├── CLAUDE.md                  # Team instructions (committed)
├── CLAUDE.local.md            # Your personal overrides (gitignored)
│
└── .claude/
    ├── settings.json          # Permissions + config (committed)
    ├── settings.local.json    # Personal permission overrides (gitignored)
    │
    ├── rules/                 # Modular instruction files
    │   ├── code-style.md
    │   ├── testing.md
    │   └── api-conventions.md
    │
    ├── skills/                # Auto-invoked workflows
    │   ├── security-review/
    │   │   └── SKILL.md
    │   └── deploy/
    │       └── SKILL.md
    │
    └── agents/                # Specialized subagent personas
        ├── code-reviewer.md
        └── security-auditor.md

~/.claude/
├── CLAUDE.md                  # Your global instructions
├── settings.json              # Your global settings
├── skills/                    # Your personal skills (all projects)
├── agents/                    # Your personal agents (all projects)
└── projects/                  # Session history + auto-memory

"Treat .claude/ as code, not configuration. The same discipline you apply to your codebase, version control, review, modularity, applies here."

Soumendra Kumar Sahoo

CLAUDE.md, the instruction manual

CLAUDE.md is the most important file in the folder. It's loaded directly into Claude's system prompt at the start of every session, which means every word in it costs tokens. This is both its power and its danger.

Keep it under 200 lines. That's not arbitrary, it's the threshold at which Claude's instruction adherence starts to degrade. Too many instructions and they start cancelling each other out.

What to put in CLAUDE.md: Run commands (npm run test), architecture decisions (Express/Prisma), strict conventions (use zod, no console.log). What to leave out: Linter configs, long theoretical explanations, rules that only apply to specific directories (those go in .claude/rules/).

There's also CLAUDE.local.md, an uncommitted override file for personal preferences that you don't want to impose on the whole team. Put your quirks there.

rules/, surgical precision at scale

As a project grows, you'll accumulate rules that only matter for specific parts of the codebase. API conventions, database access patterns, component naming, these don't belong in CLAUDE.md. That way lies bloat.

The .claude/rules/ directory solves this with path-scoped markdown files. Define paths in the YAML frontmatter, and Claude only loads those rules when you're editing matching files. Your API rules are invisible when you're working on the frontend. Zero overhead, maximum relevance.

skills/, autonomous workflows

Skills are autonomous. You place a SKILL.md inside .claude/skills/your-skill/ with a description of when to trigger it, "Use when reviewing PRs", "Use when the user asks about Excel files", and Claude reads that description, decides if it's relevant to the current task, and loads the full instructions only if it is.

agents/ and settings.json, specialisation and safety

The .claude/agents/ folder lets you define specialised personas with isolated contexts. Each agent gets its own system prompt, its own set of allowed tools, and even its own model. A "Code Reviewer" agent might be restricted to read-only tools and run on the faster, cheaper Haiku model, fast feedback, no accidental writes.

settings.json is your safety net. It controls what Claude is allowed to do at all. An allow list defines what's permitted, Bash(npm run *), Read, Write. A deny list defines what's forbidden, Bash(rm -rf *), Read(./.env). In headless or agentic runs, this is essential. Without it, you're trusting the model's judgment on operations that should require human authorisation.

Part 2: The Token Game

Why Message 30 Costs 31x More Than Message 1

Here's the thing about Claude that nobody explains at the start. Every time you send a message, Claude doesn't just read your new message. It re-reads the entire conversation history from the beginning. Every previous message. Every previous response. All of it, every single time.

This means that by the time you're on message 30, you've re-read messages 1 through 29 thirty times over. Your 30th message isn't costing you one message worth of tokens. It's costing you thirty-one message-equivalents. The meter runs exponentially, not linearly.

31× cost of message 30 vs. message 1
98.5% tokens "wasted" re-reading history in a long chat
15–20 messages before starting a fresh chat

Habit 1: Edit the prompt, don't feed the history

The single most wasteful thing you can do is send a follow-up correction. "No, I meant X." "Actually, make it shorter." "Wait, add TypeScript types." Every correction adds to the history that Claude must re-read forever after.

The fix is simple but requires a change of instinct: edit your original prompt and regenerate. Don't append, amend. Fix the source, not the symptom. This alone can cut your token usage in half on complex tasks.

Habit 2: Fresh chats are free, use them

A common mistake is treating a conversation like a continuous working session. In reality, each message you add makes the next one more expensive. After 15–20 messages, the overhead of re-reading history starts to dwarf the actual work being done.

The workflow: ask Claude to summarise where things stand, open a new chat, paste the summary, and continue. You lose nothing useful. You reset the exponential cost curve back to zero.

Habit 3: Batch your tasks

Three separate prompts means three context loads. One prompt with three tasks means one context load, and Claude produces better output because it sees the full picture at once.

"Three separate messages. Three times the overhead. One batched prompt gives you better answers at a third of the cost."

Soumendra Kumar Sahoo

Habit 4: Use Projects for recurring files

If you're uploading the same PDF, the same architecture diagram, or the same specification document at the start of every conversation, stop. Upload it to a Project once. Claude caches it. It won't re-tokenise it every session. The file becomes part of the Project's persistent memory.

Habit 5: Save your preferences in Memory

Every time you start a conversation with "Act as a senior backend developer with 10 years of experience who prefers functional programming patterns and writes clean, well-commented TypeScript..." you're burning tokens on a boilerplate you've written a hundred times. Put it in Settings → Memory. Once. Done.

Habit 6: Disable tools you're not using

Web Search, Connectors, Advanced Thinking, these features eat tokens even when you don't explicitly invoke them. Claude reads their availability and factors them in. Turn off anything you're not actively using. You can always switch them back on when you need them.

Habit 7: Right-size your model

Not every task needs the most powerful model. This is the most overlooked cost lever in the entire Claude ecosystem.

Claude Haiku

Drafts, grammar checks, formatting, summarisation. Saves 50–70% of your budget on tasks that don't need deep reasoning.

Claude Sonnet

Real coding work, medium-complexity analysis, most everyday tasks. The workhorse, reach for this by default.

Claude Opus

Deep architectural thinking, hard reasoning problems, nuanced judgment. Reserve it, don't use it for formatting a table.

Habit 8: Work with the rolling window, not against it

Claude's usage limits operate on a 5-hour rolling window, not a fixed daily reset. This means the limit renews continuously as older usage ages out. If you understand this, you can spread heavy work across morning, afternoon, and late-night sessions, and never hit a hard wall.

A practical corollary: avoid scheduling your most intensive Claude sessions during peak hours. Between 5:30 PM and 11:30 PM IST, shared infrastructure is under maximum load and limits drain faster. Front-load your heavy lifting earlier in the day.

The underlying principle: Claude is a shared resource with a token budget. Every habit here is about spending that budget on work that matters, not on overhead, not on re-reading history, not on boilerplate you've typed a thousand times.

The Golden Rules

  1. Treat tokens like currency. Edit prompts instead of appending corrections, batch multiple tasks into one message, and start a fresh chat every 15–20 messages. The meter runs exponentially, manage it deliberately.
  2. Infrastructure as prompt. Treat .claude/ as code. Keep CLAUDE.md dense and under 200 lines. Use path-scoped /rules/ for project-specific conventions so your main file stays clean.
  3. Right-size your models. Default to Haiku for lightweight tasks, escalate to Sonnet for real work, and reserve Opus only for problems that genuinely need deep reasoning. The difference in cost is dramatic.
  4. Automate what you repeat. The moment you type the same prompt twice, it belongs in a Skill, not in your clipboard. Encode your expertise once, reuse it forever.
  5. Guard the gates. Always configure settings.json with an explicit deny list for destructive commands and sensitive files. In agentic workflows, safety guardrails are not optional.
  6. Work with the rolling window. Limits renew every 5 hours, not at midnight. Spread heavy sessions across morning, afternoon, and late night. Avoid the 5:30–11:30 PM IST peak window for intensive work.
